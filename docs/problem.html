우리가 해결하는 문제
국내 웹 에이전시 PM들은 클라이언트의 추가 요구가 발생해도 비용을 청구하지 못하고 감수한다.
이유는 단순하다 — "돈 달라는 말을 꺼내기 어려워서". 이것은 도구의 부재가 아니라 구조의 부재다.

이 규모는 프로젝트 하나의 적자가 회사 현금흐름에 직접 영향을 미치지만, 전담 PM 조직과 체계적 계약 관리 프로세스를 갖추기 어려운 구간이다.

문제 구조 요약
현상원인결과
AI 도구 대중화클라이언트 기대치 상승인식 격차 심화
변경 관리 단가 기준 없음PM 개인 담력에 의존감정노동 집중
기존 PM 도구의 목적 불일치카톡+엑셀로 공백 채움일관성 없는 손실 반복
근본 원인 분석
스코프 크리프가 반복되는 데는 세 가지 구조적 원인이 있다.
원인 01
AI 도입이 만들어낸 새로운 인식 격차
AI 코딩 도구의 대중화 이후 클라이언트의 기대치가 실제 개발 비용과 벌어지기 시작했다.

Promethean Research 2024 응답자 52%가 AI를 주요 트렌드로 꼽으면서, 이 효율화가 클라이언트의 속도·가격 기대치 상승으로 이어졌다고 밝혔다.
"클라이언트들은 AI를 사용하기 때문에 더 낮은 금액을 기대하지만, 그 기대치는 고품질 결과물을 위해 실제로 필요한 인력 감소분을 반영하지 않는다."
— 시니어 어카운트 매니저, Productive.io 180개 에이전시 조사 (2025)
구조는 이렇다: 클라이언트는 "AI가 다 해주니까 싸야 한다"고 인식하고, 에이전시는 "AI 결과물 검증·커스텀 비용이 새로 생겼다"는 현실에 처한다.
원인 02
'변경 관리 단가'의 업계 표준 공백
국내 에이전시 시장에는 기능별 초기 견적 기준은 있지만, 범위 변경 발생 시 어떻게 청구하는가에 대한 표준이 없다. 기준이 없으니 협상은 PM 개인의 관계력과 담력에 전적으로 의존한다.
"제가 그러는 게 아니라, 원래 계약 범위를 벗어난 거라서요."
— PM들이 반복적으로 하는 말. 혼자 하면 아쉬운 소리, 시스템이 판정하면 협상의 성격이 바뀐다.
이것이 감정 노동이 PM에게 집중되는 구조적 이유다.
원인 03
기존 프로젝트 관리 도구는 이 문제를 해결하지 않는다
Jira, Asana, Notion 같은 도구들은 과업을 '관리'하기 위해 만들어졌다. 범위 초과를 판정하고 청구 근거를 생성하는 것은 이들의 설계 목적이 아니다.

현재 PM들은 카카오톡엑셀기억에 의존해 이 공백을 채운다. 결과는 일관성 없는 협상과 반복되는 손실이다.

Scope Manager가 제안하는 구조
PM이 "아쉬운 소리"를 하지 않아도 되는 구조를 만든다. 세 가지 핵심 역할로 작동한다.
⚖️
범위 초과의 판정자
클라이언트의 요구가 계약 범위를 벗어났는지 시스템이 기록하고 판정한다. PM의 주관이 아닌 데이터가 말한다.
📄
청구 근거의 생성자
"시스템상 과업 범위를 초과한 것으로 기록되었습니다"라는 문장을, PM이 아니라 도구가 만들어 준다.
📈
수익성의 가시화 도구
프로젝트별 실제 손익을 누적 기록하여, 다음 견적과 계약의 기준을 데이터로 만든다.
핵심 가치 제안
"시스템이 판정한 결과를 보여주면 협상의 성격 자체가 바뀐다."

기존 도구들이 과업 '관리'에 집중한 반면, Scope Manager는 관계 보호와 수익 방어에 집중한다. PM의 감정 소모 없이 계약 범위를 지킬 수 있는 구조적 장치를 제공한다.

사용 시나리오
1
계약 범위 등록 — 프로젝트 시작 시 합의된 기능 목록을 등록
2
추가 요구 기록 — 클라이언트의 추가 요구를 실시간 기록 및 자동 분류
3
범위 초과 판정 — 시스템이 계약 내/외를 자동 판정하여 리포트 생성
4
청구 근거 전달 — 감정 없이, 데이터로 추가 비용 협상 가능
MVP 우선 검증 전략
전통적인 린 스타트업 방법론을 AI 시대에 맞게 재설계했다.
⚡ 전략적 선택: 인터뷰 → MVP 순서 역전

AI 개발 도구의 발달로 MVP 제작 비용과 시간이 극적으로 낮아졌다. 이에 따라 불확실성이 높은 고객 인터뷰 단계를 생략하고, MVP를 즉시 배포하여 실제 사용자의 행동 데이터로 가설을 검증하는 전략을 선택했다.

전통적 린 스타트업
  • 고객 인터뷰 설계·진행
  • 문제 검증 (말에 의존)
  • 솔루션 검증
  • MVP 개발·배포
  • 행동 데이터 수집
→
AI 시대 역전 전략
  • MVP 즉시 개발·배포
  • 실제 사용/이탈 패턴 관찰
  • 행동 데이터로 가설 검증
  • 데이터 기반 인터뷰 (선택적)
  • 검증된 기능 확장
전략의 핵심 논리
🗣️
고객 인터뷰의 한계말(intention)과 행동(behavior)이 불일치하는 경우가 많다. "쓸 것 같다"와 "실제로 쓴다"는 다르다.
📊
MVP 행동 데이터의 우위실제 사용/이탈 패턴이 인터뷰보다 더 정직한 신호를 제공한다.
💰
비용 역전 현상AI 덕분에 MVP 개발 비용이 인터뷰 설계/진행 비용보다 낮아진 역전 현상이 발생했다.
⚡
결론"만들어서 보여주는" 것이 "물어보는" 것보다 더 빠르고 정확한 검증 방법이 되었다.

✓ 범위 등록 완료율
✓ 첫 추가 요구 기록까지의 시간

✓ 2주 후 재방문율

✓ 청구 근거 리포트 생성 횟수
미검증 가설 — 열린 질문들
이 문서는 현재까지 확인된 사실과 구조적 논리를 바탕으로 작성되었다. 아래 질문들은 MVP 행동 데이터와 선택적 인터뷰를 통해 반드시 검증해야 할 열린 과제다.
검증 우선순위 — 핵심 가설
❓
추가 비용 청구 포기율 5~20인 에이전시 PM 중 실제로 추가 비용 청구를 포기하는 비율은 얼마인가? 행동 데이터로 검증 가능
❓
포기의 실제 이유 포기의 주요 이유는 증거 부족인가, 아니면 관계 악화에 대한 우려인가? 이탈 지점 분석 필요
❓
시스템 판정 수용도 클라이언트는 'AI 또는 시스템 판정 결과'를 수용하는가, 아니면 반발하는가? 리포트 전달 후 피드백 추적
❓
경쟁 위협 — 기존 도구의 확장 기존 협업 도구(Jira, Notion 등)가 이 기능을 추가할 경우 차별점이 유지되는가? 시장 감시 필요
