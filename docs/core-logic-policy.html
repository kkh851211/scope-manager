v1.0 Core Logic & Policy 데이터 정책 & 로직

핵심 로직 전체 흐름

사용자 입력 → AI 처리 → 피치덱 생성 → 데이터 저장의 엔드-투-엔드 정책 개요

5단계 핵심 로직 파이프라인
STEP 1 — 컨텍스트 수집
온보딩 데이터로 AI 산정 기반 구성
가입 시 수집한 직군·회사 규모·재직기간이 모든 AI 호출의 시스템 컨텍스트로 주입된다. 이 데이터 없이는 금액 산정이 불가능하므로 온보딩 3문항은 필수(Required)로 처리된다. 직군별 M/M 단가 추정 테이블을 내부적으로 유지하며 프롬프트에 반영한다.
필수 3문항MM 단가 매핑프롬프트 컨텍스트 주입
STEP 2 — 이슈 입력 & 검증
텍스트 입력 유효성 검사 및 전처리
이슈 텍스트는 최소 50자 / 최대 500자 제한으로 LLM 컨텍스트 낭비와 빈 입력을 동시에 방지한다. 입력 저장 시 개인식별정보(이름·연락처) 패턴을 감지해 마스킹 경고를 표시한다. 임시저장은 30초마다 자동 실행해 데이터 유실을 방지한다.
50~500자 검증PII 패턴 감지30초 자동저장
STEP 3 — AI 구조화 & 금액 환산
LLM 호출 → JSON 파싱 → 스키마 검증
LLM 응답은 반드시 Zod 스키마로 검증한다. 필수 필드(impact_low, impact_high, type, calc_basis) 중 하나라도 누락 또는 타입 불일치 시 자동 1회 재시도. 2회 연속 실패 시 사용자에게 재시도 UI 표시. AI 원본값은 절대 덮어쓰지 않고 별도 컬럼에 보존한다.
Zod 스키마 검증원본값 불변 보존재시도 1회 자동
STEP 4 — Human-in-the-loop 수정
사용자 수정값 저장 & 편차 로깅
사용자가 금액을 수정하면 ai_impact_low/high(원본)와 user_impact_low/high(수정값)를 분리 저장한다. 수정 시마다 edit_delta_pct를 계산해 이벤트로 로깅한다. 이 데이터가 H2 가설(AI 수용률) 검증의 핵심 근거가 된다.
원본/수정값 분리 저장delta_pct 로깅H2 가설 근거
STEP 5 — 피치덱 생성 & 공유
스냅샷 저장 → PDF 렌더 → 공유 토큰 발급
피치덱 생성 시점의 데이터를 불변 스냅샷(immutable snapshot)으로 저장한다. 이후 원본 이슈를 수정해도 생성된 피치덱은 변경되지 않는다. 공유 링크에는 UUID 기반 share_token을 발급해 추측 불가 URL을 보장한다. 공유 뷰어에서는 이름·회사명 등 민감 정보를 마스킹할 수 있는 옵션을 제공한다.
불변 스냅샷UUID share_token민감정보 마스킹 옵션
로직 설계의 3대 원칙
🔒
원칙 1 — AI 원본 불변성
AI가 산출한 원본값은 절대 덮어쓰지 않는다. 사용자 수정값은 별도 컬럼에 저장. 이것이 H2 가설 측정의 데이터 무결성 기반.
👤
원칙 2 — 데이터 주권은 사용자에게
이슈 텍스트·금액은 본인만 볼 수 있다. 피치덱 공유는 사용자의 명시적 행동으로만 발생. 기본값은 비공개.
📊
원칙 3 — 행동 데이터 완전 수집
가설 검증에 필요한 행동 이벤트는 서버·클라이언트 이중 수집으로 누락률 0% 목표. 텍스트 내용은 수집하지 않음.

임팩트 금액 산정 로직

비공식 기여를 재무적 절감액으로 변환하는 핵심 알고리즘

이슈 유형별 산정 공식
🛡️ 사전 예방 (Proactive Prevention)
방어액 = M/M 단가 × 예상 지연 기간(일) × 팀 규모 보정
사고가 발생했을 경우의 기회비용·재작업 비용을 역산. 예: "2주 일정 지연을 막았다" → 팀 5인 × 2주 M/M 비용. Confidence: 이슈 구체성에 따라 high/medium/low.
🔥 위기 수습 (Reactive Resolution)
방어액 = 장애 영향 규모 × 시간당 손실 단가 × 수습 시간
실제 발생한 문제를 해결한 경우. 서비스 중단 시간·매출 손실·페널티 회피 기준. 서비스 규모·업종별 단가 테이블 참조. 구체적 수치 언급 시 Confidence 상승.
🧩 역할 중첩 (Role Overlap)
방어액 = (해당 직무 외주 단가 - 현 M/M 비용 분) × 기간
본업 외 업무를 수행한 기여. 외주·채용 대비 비용 절감액으로 환산. 예: "CS를 3개월 겸직" → CS 담당자 채용 또는 외주 비용 대비 절감분.
🤝 관계·조율 (Coordination Value)
방어액 = 협상 성과 금액 × 기여 기여도 추정 (30~70%)
계약 성사·파트너십·갈등 해소 등 간접 재무 기여. 기여도 추정에 불확실성이 높아 항상 Confidence: medium 이하. 범위(하한~상한)를 넓게 제시.
직군별 M/M 단가 추정 테이블 (내부 기준 · 국내 SME 50인 미만)
직군월 평균 인건비 범위일 단가 환산시간 단가 환산적용 기준

⚠️ 이 테이블은 AI 산정의 초기 기준값이며, 실사용자 수정 데이터(edit_delta_pct)가 축적될수록 직군별 보정계수가 고도화된다. v1.0은 보수적 하한값을 기본으로 사용해 과장 위험을 줄인다.
Confidence 레벨 판정 기준
HIGH 높은 신뢰도
이슈 텍스트에 구체적 수치(금액·기간·팀 규모)가 포함되어 있거나, 명확한 결과(계약 성사·장애 복구 시간)가 언급된 경우.
예: "A 클라이언트 계약 해지 위기를 2일 내 조율해 3천만원 계약 유지"
→ impact_low: 2,500만, impact_high: 3,000만, confidence: "high"
MEDIUM 중간 신뢰도
이슈 유형은 명확하나 수치 근거가 간접적인 경우. "몇 주", "팀 전체" 등 상대적 표현만 있어 M/M 단가 추정이 필요한 상황.
예: "출시 2주 전 치명적 버그를 발견해 일정 지연을 막음"
→ impact_low: 300만, impact_high: 800만, confidence: "medium"
LOW 낮은 신뢰도
이슈 설명이 추상적이거나, 임팩트 규모를 추론하기 어려운 경우. 또는 직군 미선택으로 M/M 단가 기준값이 없는 경우. 사용자 직접 수정을 적극 유도.
예: "팀 분위기가 나빠질 뻔한 상황을 잘 중재했다"
→ impact_low: 50만, impact_high: 300만, confidence: "low" + 수정 유도 배너 표시

스냅샷 정책
생성 시점의 이슈 데이터를 snapshot_data에 통째로 JSON 저장. 이후 원본 이슈 수정·삭제와 무관하게 피치덱 내용은 고정. "협상 당일 제출한 문서"의 신뢰성 보장이 목적.
공유 정책
기본값 is_shared = false. 사용자가 명시적으로 공유 활성화 시에만 share_token으로 접근 가능. 공유 해제 즉시 링크 접근 차단. 공유 뷰어에서 민감 정보 마스킹 옵션 제공.

이중 수집 정책
클라이언트 SDK(Mixpanel) + 서버 사이드 API(/api/track) 동시 전송. 광고 차단기로 클라이언트 수집 실패 시 서버 수집이 백업. 중요 이벤트(pitchdeck_generated, signup)는 서버 필수.
프라이버시 경계
이벤트 properties에 텍스트 내용·금액 절대값 저장 금지. edit_delta_pct(%)처럼 상대값·메타데이터만 수집. 분석 목적 최소화 원칙.
DB 트리거 & 제약 조건 명세
-- 1. AI 원본 필드 불변 트리거 CREATE OR REPLACE FUNCTION prevent_ai_field_update() RETURNS TRIGGER AS $$ BEGIN IF OLD.ai_impact_low IS NOT NULL AND (NEW.ai_impact_low != OLD.ai_impact_low OR NEW.ai_impact_high != OLD.ai_impact_high) THEN RAISE EXCEPTION 'AI original fields are immutable'; END IF; RETURN NEW; END; $$ LANGUAGE plpgsql; -- 2. edit_delta_pct 자동 계산 트리거 CREATE OR REPLACE FUNCTION calc_edit_delta() RETURNS TRIGGER AS $$ BEGIN IF NEW.user_impact_low IS NOT NULL THEN NEW.edit_delta_pct := -- (user_mid - ai_mid) / ai_mid * 100 ROUND(((NEW.user_impact_low + NEW.user_impact_high) / 2.0 - (OLD.ai_impact_low + OLD.ai_impact_high) / 2.0) / NULLIF((OLD.ai_impact_low + OLD.ai_impact_high) / 2.0, 0) * 100, 1); NEW.is_ai_accepted := FALSE; END IF; RETURN NEW; END; $$ LANGUAGE plpgsql; -- 3. RLS: 본인 데이터만 접근 ALTER TABLE issues ENABLE ROW LEVEL SECURITY; CREATE POLICY own_issues ON issues USING (user_id = auth.uid());

LLM 전송 데이터 처리 정책
⚠️ 핵심 위험: 이슈 텍스트가 OpenAI API로 전송된다
실무자가 기술한 이슈 내용이 OpenAI 서버로 전송된다. 개인정보처리방침에 이 사실을 명시하고, 이슈 입력창 하단에 "AI 분석을 위해 OpenAI로 전송됩니다" 고지 문구를 반드시 표시해야 한다.
전송 범위이슈 raw_text + 직군·규모 컨텍스트. 이름·회사명·연락처는 전송 금지 (입력 전 PII 감지 경고).
OpenAI 정책OpenAI API 사용 시 데이터를 모델 학습에 사용하지 않음 (Zero Data Retention 옵션 활성화 권장).
고지 방식입력창 하단 상시 표시 + 첫 분석 시 명시적 동의 체크박스 (opt-in).
민감 정보 필터전송 전 이름(홍길동 패턴)·전화번호·주민번호·계좌번호 정규식 감지 후 [개인정보]로 마스킹.
서버 로그Edge Function 서버 로그에 raw_text를 기록하지 않음. request_id와 메타데이터(토큰 수·레이턴시)만 로깅.
위반 시 대응개인정보 침해 신고 시 72시간 내 KISA 신고 의무. 개인정보처리방침에 DPO 연락처 명시.
사용자 권리 보장 (PIPA 제35~39조)
📥 열람권 · 이동권
사용자는 언제든 자신의 이슈 데이터·피치덱·이벤트 메타데이터를 JSON 형태로 내보내기(Export) 요청 가능. 요청 후 3영업일 내 제공.
🗑️ 삭제권 (잊힐 권리)
탈퇴 또는 개별 이슈 삭제 요청 시 30일 내 완전 삭제 (백업 포함). 단, 법령 의무 보존 데이터(결제 기록 등) 제외. 삭제 완료 시 이메일 통보.
✏️ 정정권
온보딩 정보(직군·규모·재직기간) 및 수정 가능한 이슈 필드는 앱 내에서 직접 수정 가능. AI 원본 필드는 불변 정책으로 수정 불가 (단, 해당 이슈 전체 삭제 후 재입력 가능).
🚫 처리 정지권
행동 이벤트 수집에 대해 동의 철회 가능. 철회 시 이후 이벤트 수집 중단 (단, 핵심 서비스 기능은 유지). 설정 페이지에서 온/오프 토글 제공.

⏱️ 공유 만료 옵션
선택적 만료 설정: 7일·30일·무기한. 만료 후 링크 접근 시 "만료된 링크입니다" 페이지 표시. 재발급 가능.
🚫 공유 즉시 해제
is_shared = false 설정 즉시 링크 접근 불가. 토큰 자체는 유지 (재활성화 가능). DB 레코드 삭제가 아닌 플래그 전환.

증빙 이미지 (Supabase Storage)
피치덱 삭제 시 연동 삭제
보존 기간

orphan 이미지(연결된 이슈 없음) 주 1회 자동 스캔 후 삭제. Supabase Edge Function으로 크론 구현.
용량 관리
업로드 시 WebP 변환 + 1MB 압축. 유저당 총 50MB 상한. 초과 시 오래된 이미지 삭제 권고.

탈퇴 시 처리
탈퇴 즉시 user_id를 익명 UUID로 교체. 이벤트 내용은 보존하되 개인 식별 불가 처리. 집계 통계 무결성 유지.
가설 검증 데이터
H1~H5 가설 검증에 사용되는 집계값(CVR, 수용률 등)은 익명화 후 무기한 보존. 제품 의사결정의 역사적 근거로 활용.
자동 삭제 파이프라인 (Supabase Cron)
-- 매일 03:00 실행: 논리 삭제 후 7일 경과 이슈 물리 삭제 SELECT cron.schedule( 'cleanup-deleted-issues', '0 3 * * *', $$ DELETE FROM issues WHERE is_deleted = TRUE AND deleted_at < NOW() - INTERVAL '7 days'; -- 연결 없는 orphan 이미지 삭제 (Storage 정리) DELETE FROM storage.objects WHERE bucket_id = 'evidence' AND name NOT IN ( SELECT UNNEST(evidence_urls) FROM issues WHERE is_deleted = FALSE ); $$ ); -- 매주 월 04:00: 비활성 계정(12개월) 경고 이메일 트리거 SELECT cron.schedule( 'warn-inactive-users', '0 4 * * 1', $$ SELECT send_warning_email(id) FROM users WHERE last_login < NOW() - INTERVAL '12 months' AND warned_at IS NULL; $$ );
