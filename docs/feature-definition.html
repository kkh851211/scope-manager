기능 정의서 & 백로그
Scope Manager의 전체 기능을 정의하고 개발 우선순위를 설계합니다.
MVP는 가설 검증에 필요한 최소한의 기능만 포함하며, 검증 결과에 따라 백로그를 동적으로 조정합니다.
⚡ 기능 정의 원칙 — 검증 최우선

모든 기능은 "어떤 가설을 검증하는가?"를 기준으로 우선순위를 결정합니다. 고객 인터뷰 대신 MVP를 배포하고 행동 데이터로 검증하는 전략에 따라, MVP에는 H1(문제 가설)·H2(솔루션 가설) 검증에 직결되는 기능만 포함합니다.

⚖️ 영역 A — 범위 관리 (핵심)
계약 범위 등록 → 추가 요구 기록 → 범위 초과 자동 판정. MVP의 핵심 가치를 전달하는 기능 묶음.
📄 영역 B — 리포트 & 청구 (AHA 모멘트)
범위 초과 판정 리포트 자동 생성 → PM의 감정 노동 없이 청구 근거 전달. H2 가설 검증의 핵심.
📈 영역 C — 수익성 분석 (Paid 가치)
프로젝트별 실시간 손익 추적 → 견적 데이터 누적. Paid 플랜의 핵심 차별 기능.
기능 정의 원칙 (4가지)
  • 가설 연결성 — 모든 기능은 검증할 가설 번호(H1~H4)를 명시
  • AHA 경로 최단화 — 가입→AHA 모멘트까지 클릭 수 최소화
  • PM 감정 노동 제거 — 사람이 말해야 할 것을 시스템이 대신 말함
  • 데이터 누적 구조 — 사용할수록 다음 견적의 정확도가 높아지는 선순환

F-01 · 영역 A · H1 검증
프로젝트 생성 & 계약 범위 등록
MVP 필수 개발 예정
PM이 프로젝트를 생성하고 계약 범위(기능 목록 + 예상 공수)를 등록하는 기능. 이후 모든 범위 판정의 기준선이 된다. 온보딩 설문(포기 경험 2문항)도 이 단계에서 수집한다.
유저 스토리: PM으로서, 새 프로젝트 시작 시 클라이언트와 합의한 기능 목록을 등록하고 싶다. 그래야 나중에 추가 요구가 생겼을 때 "이건 계약 범위 밖"이라고 시스템이 판정해줄 수 있기 때문이다.

F-02 · 영역 A · H1 검증
추가 요구 실시간 기록
MVP 필수 개발 예정
클라이언트의 추가 요구를 발생 즉시 기록하는 기능. 날짜·내용·요청 채널(카카오톡/이메일/전화)을 기록한다. 빠른 입력을 위해 모바일 친화적 UI가 필요하다. 기록 시 자동으로 계약 범위와 비교를 시작한다.
유저 스토리: PM으로서, 클라이언트가 카카오톡으로 추가 기능을 요청할 때 즉시 기록하고 싶다. 나중에 "그런 말 했나요?"라는 상황을 방지하기 위해.

F-03 · 영역 A · H2 검증
범위 초과 자동 판정 엔진
MVP 필수 개발 예정
등록된 계약 범위와 추가 요구를 비교하여 "범위 내 / 범위 초과 / 판단 필요" 3단계로 자동 분류하는 핵심 엔진. AI(Claude API)를 활용하여 텍스트 의미 기반 유사도 비교를 수행한다. 이것이 Scope Manager의 핵심 차별점이다.
유저 스토리: PM으로서, 클라이언트의 추가 요구를 입력하면 시스템이 자동으로 계약 범위를 벗어났는지 판정해주길 원한다. "제가 아니라 시스템이 그렇게 판정했어요"라고 말할 수 있게.

F-04 · 영역 B · H2 검증 (AHA)
범위 초과 판정 리포트 자동 생성
MVP 필수 개발 예정
"시스템상 과업 범위를 초과한 것으로 기록되었습니다"라는 문장을 포함한 공식 리포트를 자동 생성하는 기능. PM의 감정 노동 없이 클라이언트에게 전달 가능한 형태로 출력된다. 이것이 핵심 AHA 모멘트다.
유저 스토리: PM으로서, 범위 초과가 판정되면 클라이언트에게 바로 보낼 수 있는 공식 문서를 자동으로 받고 싶다. 내가 직접 말하는 게 아니라 시스템 문서로 전달하고 싶다.

영역 A — 범위 관리
Epic A1
계약 범위 기준선 수립
US-A1-01: 에이전시 PM으로서, 프로젝트 시작 시 클라이언트와 합의한 기능 목록을 5분 이내에 등록하고 싶다. 그래야 이후 추가 요구와 비교할 수 있기 때문이다.
✓ 기능 항목 + 공수 입력 ✓ 5분 이내 완료 ✓ 저장 후 즉시 수정 가능
US-A1-02: 대표 겸직 PM으로서, 기존 계약서 PDF를 업로드하면 기능 목록이 자동 추출되길 원한다. 수동 입력 시간을 줄이고 싶기 때문이다.
⚠️ Phase 2 기능 — MVP에서는 수동 입력으로 대체
Epic A2
추가 요구 캡처
US-A2-01: PM으로서, 클라이언트가 카카오톡으로 요청을 보내는 즉시 앱에 기록하고 싶다. 나중에 분쟁이 생겼을 때 증거로 쓰기 위해.
✓ 모바일 원탭 입력 ✓ 날짜 자동 기록 ✓ 스크린샷 첨부
US-A2-02: PM으로서, 추가 요구를 입력하자마자 "이게 계약 범위 안인지 밖인지" 바로 알고 싶다. 대화 중에 빠르게 판단해야 하기 때문이다.
✓ 입력 후 3초 이내 판정 ✓ 판정 근거 텍스트 표시 ✓ 오버라이드 가능
영역 B — 리포트 & 청구
Epic B1
청구 근거 자동 생성 (AHA 모멘트)
US-B1-01: PM으로서, 범위 초과 판정이 나면 클라이언트에게 바로 보낼 수 있는 공식 문서를 받고 싶다. "제가 그러는 게 아니라 시스템이 판정했어요"라고 말할 수 있게.
✓ 자동 리포트 생성 ✓ PDF 다운로드 ✓ 링크 공유
US-B1-02: PM으로서, 리포트를 앱에서 직접 클라이언트 이메일로 발송하고 싶다. 첨부파일 다운로드 없이 바로 전달하기 위해.
⚠️ Starter 이상 기능
US-B1-03: PM으로서, 추가 청구 금액을 리포트에 포함시키고 싶다. "범위 초과이므로 N만원이 추가됩니다"라는 문장까지 자동 생성되길 원한다.
⚠️ Phase 2 — MVP에서는 금액 입력 후 포함만 지원
영역 C — 수익성 분석
Epic C1
프로젝트 손익 가시화
US-C1-01: 에이전시 대표로서, 진행 중인 프로젝트의 현재 수익성을 실시간으로 보고 싶다. 프로젝트가 끝난 후에야 적자인 걸 알게 되는 상황을 없애고 싶다.
✓ 계약금 · 투입시간 · 추가청구 기반 실시간 손익 표시
US-C1-02: PM으로서, 이번 프로젝트에서 발생한 모든 추가 요구와 청구 결과를 한 페이지에서 보고 싶다. 다음 프로젝트 견적의 기준으로 쓰기 위해.
✓ 추가 요구 목록 · 판정 결과 · 청구 성공 여부 집계
